Hoje quando falamos sobre técnologias de desenvolvimento nós não mais falamos sobre sistemas operacionais, servidores web específicos, linguagem de backend ou banco de dados.

A stack JAM não é sobre uma tecnologia específica. É uma nova maneira de construir websites e aplicativos que entregam uma melhor perfomance, segurança e baixo custo de escalabilidade, e uma melhor experiência de desenvolvimento.

### Afinal, o que é JAM stack?

Você já deve ter visto ou trabalhado em um site com a stack JAM. Ele não precisa incluir todos os atribuitos do Javascript, API`s e Markup. Ele precisa ser contruido a mão ou com Jekyll, Hugo, Next, Gatsby ou qualquer outro gerador de site estático.

A principal característia de um site com a stack JAM é que ele não depende de um servidor web.

#### Javascript

Qualquer programação dinâmica durante o ciclo de requisição/resposta é lidado pelo Javascript, rodando apenas no browser. Isso pode ser em qualquer framework, biblioteca ou mesmo vanilla Javascript.

#### APIs

Todos os processos no lado do servidor e ações do banco de dados são abstraídas em API’s reutilizáveis, acessadas via HTTPS com JavaScript. Elas podem ser personalizadas ou aproveitar serviços externos.

#### Markup

Templates de marcação deve ser pré-construida no momento do deploy, geralmente usando algum gerador de site para seu conteúdo ou uma ferramenta de criação para aplicações web.

### Quando um site não é construído com a stack JAM?

Qualquer site que dependa de uma ligação entre o browser e o servidor não é construido com a stack JAM. Isso inclúi:

- Um site contruído com um gerenciador de conteúdo no lado do servidor, como WordPress, Drupal, Joomla ou Squarespace.
- Um monólito que rode no servidor que dependa da linguagem Ruby, Node ou qualquer outra.
- Uma aplicação de página única que usa renderização isomórfico para construir suas telas de forma runtime no servidor.

### Porque usar a stack JAM?

#### Melhor Perfomance

Porque esperar por páginas serem construídas na hora quando você pode gera-las no deploy? Quando se trata de minimizar o tempo de renderização, nada supera a distribuição de arquivos pré construídos via CDN.

#### Alta Segurança

Com a abstração de processos do servidor em API’s de micro serviços, áreas que podem ser atacadas são reduzidas. Você também pode aproveitar a experiência de serviços de domínios externos.

#### Barato e Fácil de Escalar.

Quando os arquivos do seu deploy podem ser servidos em qualquer lugar, escalar sua aplicação se torna uma questão de servir esses arquivos em mais lugares. CDN’s é perfeito pra isso e frequentemente incluem escalabilidade em seus planos.

#### Melhor Experiência no Desenvolvimento

Controlar a junção e separação de responsabilidades permite um desenvolvimento e debug mais preciso e o aumento de opções de CMS’s disponíveis atualmente remove a necessidade de manter uma tecnologia para conteúdo e outra para marketing.
